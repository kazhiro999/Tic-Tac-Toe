# kintone-ui の機能追加／デザイン改善フロー

## このドキュメントのコンセプト

- 誰に: kintone-uiのデザイン／開発に携わるメンバーに
- 何を: 仕様作成からリリースまでの進め方がイメージできた

と言ってほしい。

下記に示す流れに沿って、kintone-uiに機能追加／デザイン改善を行います。<br/>
ここでいうデザインとは、見た目の変更、キーボード操作の追加など、広い意味でのデザインです。

## （前提）進め方のパターン

機能追加／デザイン改善するために、大きくわけて仕様検討フェーズ、実装フェーズの2フェーズがあります。<br/>
仕様検討後、関係者で合意できたら実装フェーズに進みます。<br/>
進め方として、kDSチーム（kintone Design Systemチーム）だけで進めることもあれば、外部チームと協力・分担して進めることもあります。

下記に進め方のいくつかのパターン例を示します。

| パターン                                   | 仕様検討フェーズ                                 | 合意関係者                                                         | 実装フェーズ                                       |
| ------------------------------------------ | ------------------------------------------------ | ------------------------------------------------------------------ | -------------------------------------------------- |
| kDS チーム主体                             | kDS チームメンバーで検討                         | sakito さん＋仕様検討した kDS チームメンバー                       | kDS チームで実装                                   |
| 実装を外部チームに委託                     | kDS チームメンバーで検討                         | sakito さん＋仕様検討した kDS チームメンバー                       | 外部チームに委託してレビューは kDS チーム          |
| 機能リクエスト（kDS チーム主体）           | kDS チームで検討                                 | sakito さん＋仕様検討した kDS チームメンバー＋リクエストしたチーム | kDS チームで実装                                   |
| 機能リクエスト（リクエストしたチーム主体） | リクエストしたチームが行い、kDS チームでレビュー | sakito さん＋リクエストしたチーム＋ kDS チーム                     | リクエストしたチームが実装し、kDS チームでレビュー |

## 1. 機能追加／デザイン改善の計画を sakito さんと合意する

kDSチームは、デザイン戦略の意思決定をしているsakitoさんに、機能追加／デザイン改善の計画を相談します。<br/>
[#kintone-design](https://cybozu.slack.com/archives/C02N891C4CE)で@sakitoさん、もしくは@kintone-design-teamにメンションするか、kintone Design Teamの朝会で共有します。<br/>
sakitoさんやデザイナーと議論して、懸念点を洗い出した上で、計画を進めることを合意します。<br/>
なおkintone Design Team朝会は毎日10:35から[#kintone-design](https://cybozu.slack.com/archives/C02N891C4CE)のハドルミーティングで行っています。

## 2. 仕様検討フェーズ

仕様検討のissueを積みます。<br/>
仕様検討では、下記のことを行います。

- ドキュメントの作成
  - 仕様検討用のブランチをきります
  - Storybookの各コンポーネントのdocsを編集します
  - プルリクエストを作成し、ドキュメントのレビューをkDSチームメンバーにお願いします
- Figmaの作成
  - 仕様検討用のブランチをきります
  - プロトをFigmaで再現します
  - kDSチームメンバーにレビューをお願いします

ブランチはマージせず、3.の手順に進みます。

## 3. 検討した仕様を関係者間で合意する

作成したドキュメントとFigmaをもとに、関係者間で議論し、合意します。

関係者とは：

- sakitoさん
- 仕様検討したkDSチームメンバー
- 機能リクエストの場合は、リクエストしたチームメンバー

フィードバックをうけて調整します。<br/>
合意できたら、ドキュメントとFigmaをマージして、4.に進みます。

## 4. 実装フェーズ

実装のissueを積みます。<br/>
実装では下記のことを行います。

- 実装
  - 検討した仕様を実装します
- テスト実装
  - ドキュメントに記載した仕様をもれなくテストします
- レビュー
  - kDSチームメンバーにレビューをお願いします
- 試験
  - 検討した仕様どおりに実装されているか、動作するかを確認します

## 5. スプリントレビューで共有

kintoneスプリントレビューで共有します。<br/>
スプリントレビューは毎週水曜日の15:00~16:00開催です。<br/>
スプリントレビューの実況スレに共有する内容を事前に書き込みましょう。

## 6. リリース

kDSチームでは、毎週木曜日にブランチフリーズし、金曜日にリリースをしています。<br/>
リリースのバージョンは、[バージョニングポリシー](?path=/docs/contributing-バージョニングポリシー--docs)に従います。<br/>
リリース後、必要に応じて外部チームにkintone-uiの取り込みをお願いしたり、取り込み用のプルリクエストをkDSチームで作成して渡したりします。
